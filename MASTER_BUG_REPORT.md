**Date**: 2026-07-01
**Method**: 113 findings from 12 independent full-file-read audit passes (deduplicated below to ~85 unique items), plus findings I confirmed myself by directly running the pytest suite, running `npm run build`, curling live production URLs, and querying the real Vercel API. Every item cites a file and, where applicable, a line number. Items independently flagged by more than one audit pass are marked **(confirmed independently ×N)** — treat these as higher-confidence.

Severity follows the audit's own classification: **Critical** = data loss, security exposure, or a claim that is provably false; **High** = a feature that does not do what it appears to do, or a bug that produces wrong output silently; **Medium** = a real defect with a narrower blast radius; **Low** = code-quality/hygiene debt.

## Cross-cutting pattern worth naming explicitly

Nearly every "Critical" and "High" finding above shares one shape: **a mock/placeholder implementation that is indistinguishable from a real one at the API/UI boundary**, combined with **prior self-verification reports that tested the shape of the response, not its provenance** (status code 200 + plausible-looking JSON = PASS, regardless of whether the data came from a real integration or a hardcoded literal). Any fix plan needs to close both gaps — replace the mock, and make sure whatever verifies it next can tell the difference.
